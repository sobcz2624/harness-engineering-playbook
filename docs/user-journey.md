# User Journey & User Stories（第二版）

> **核心设计原则：**
> 1. 每条用户故事都有"已经……"的前置状态，用于检验章节依赖链
> 2. 剔除了为特定内容捏造的故事，只保留真实痛点
> 3. 用用户的语言，不用作者的语言
> 4. 分卷结构，每卷有卷级用户故事定义读者状态和目标
> 5. 演进作为每卷内的独立子级（不确定是否独立成章，但一定是卷的组成部分）
> 6. 组织变革卷使用独立 persona

---

## 用户画像

### 主画像：小张

28 岁，3 年经验的全栈工程师，团队 8 人。用 Cursor / Claude Code 半年了。体感效率大概 1.5x~2x。隐隐觉得自己用法有问题，但说不清楚问题在哪。

### 副画像：老王（卷三使用）

35 岁，Tech Lead / 技术负责人。管一个 10-15 人的团队。团队里有几个人已经在用 Agent，产出参差不齐。正在思考怎么在团队层面推 Agent。

---

## 引言（不在本文档展开）

> 引言负责两件事：
> 1. **共鸣建立：** 用一组症状描述做读者筛选——"你是不是经常跟Agent来回改四五轮"、"你是不是做完才发现各种漏洞"、"你是不是觉得自己只有1.5x"……让目标读者确认"这本书是写给我的"。
> 2. **框架引入：** 诊断这些症状的共同根因——vibe coding 是开环的，harness engineering 是闭环的。给出全书的地图，但不展开具体怎么做。
>
> 引言的具体内容和结构另行设计，不属于用户故事文档的范围。

# 卷一：从 1.5x 到 5-10x —— 个人基础

> **已经：** 用 Agent 写代码有一段时间了，知道 vibe coding 不够，读了引言理解了 harness engineering 的大方向。
> **但是：** 具体到每天的工作，我写的需求 Agent 总是理解不对，产出的代码我也不知道怎么验证，整体效率大概 1.5x。
> **我想读完这一卷后能：** 写出 Agent 能准确执行的输入，建立可靠的验证闭环，单次交互效率到 5-10x。

---

## 阶段一：把输入说清楚

> 小张在这个阶段的状态：读了引言，理解了 harness engineering 的基本思路——不能光靠 prompt 试出来，需要给 Agent 明确的输入、可验证的产出。大方向有了，但具体到每天的工作，他不知道第一步该改什么。

### 1.1 来回改

**已经：** 每天用 Agent 写代码，确实比自己从零写快。
**但是：** 经常一个功能要跟 Agent 来回改三四轮才勉强能用。有时候改到第四轮还不如自己直接写。
**我想知道：** 到底是它不够聪明，还是我说需求的方式有问题？

### 1.2 改了几轮发现方向就是错的

**已经：** 开始注意写更具体的需求了，细节上的来回少了一些。
**但是：** 有时候跟 Agent 改了三四轮细节，最后才发现方向根本就是错的。比如我说"加一个用户搜索功能"，它上来就接了 Elasticsearch，但我们就是个内部管理后台，SQL LIKE 就够了。前面几轮我都在纠结搜索结果的排序逻辑，第四轮才反应过来——整个技术选型就不对。早干嘛去了。
**我想知道：** 怎么在一开始就把大方向对齐，别等改了半天才发现白费力气？

### 1.3 做出来才发现漏了需求

**已经：** 学会了先把大方向、技术选型这些讲清楚再让 Agent 动手，返工少了很多。
**但是：** Agent 做完一个功能，我自己打开看了一眼就发现不对——列表没有分页，长了就一直往下拉；设置页面没有鉴权，任何人都能直接访问。这些不是 Agent 做错了，是我根本没提。我说了"做一个设置页面"，但"设置页面需要鉴权"这种事我觉得不言自明，就没写。Agent 不会自己补这些。好在这次是我自己看的时候发现的，没上线。但每次都要做完再人肉检查才能发现漏了什么，效率也不高。
**我想知道：** 有没有办法在交给 Agent 之前就系统地检查我的需求是否完备，而不是每次做完再一个个找漏洞？

---

## 阶段二：知道它做的对不对

> 小张在这个阶段的状态：输入问题改善了不少，Agent 单次产出靠谱多了，效率确实提升了。他开始信任 Agent 的产出，review 也越来越粗——"反正之前几次都挺好的"。**然后出事了。**

### 2.1 上线炸了才知道有 bug

**已经：** spec 写得比以前清楚多了，Agent 产出的代码方向基本正确，最近几个功能上线都挺顺利。我开始信任它，review 也没那么仔细了。
**但是：** 上周上线后出了一个生产事故。查下来发现 Agent 写的代码在正常路径上完全没问题，但有一个竞态条件没处理。回头看，这种问题我自己写代码时也会凭经验注意到，但 Agent 不会——它只管把我描述的功能实现了。
**我想知道：** 我以为"输入写好了就够了"，但显然不是。Agent 产出的代码，我到底该怎么验？

### 2.2 让它自己写测试，自己出题自己答

**已经：** 上次事故后我意识到需要验证。最直接的想法——让 Agent 自己写测试。
**但是：** 测试全绿。我不放心，自己手动试了几个边界情况，又发现 bug 了。仔细看 Agent 写的测试才明白——它是看着自己的实现写的测试，当然能全过。它不会测"代码应该做什么"，只会测"代码做了什么"。
**我想知道：** 测试要怎么组织，才不是自己出题自己答？

### 2.3 看了也看不出来

**已经：** 知道不能让 Agent 自测了，我开始自己认真 review 每个 PR。
**但是：** Agent 的代码表面上很干净，跑起来也没问题。但上线后又出 bug 了。回头仔细看才发现：它在一个地方 hard code 了一个返回值，另一个地方直接 catch 了所有异常然后静默吞掉了，还有一个地方用了一个 setTimeout 绕过了竞态条件而不是真正解决它。这些不是粗心能发现的，是 Agent 在"用最省力的方式让代码跑通"。逐行看都不一定能发现，何况我一天要看十几个 PR。
**我想知道：** 人肉 review 靠不住，Agent 自测也靠不住，到底用什么方式才能可靠地抓住这些问题？

### 2.4 等做完再查就晚了

**已经：** 开始投资自动化验证——写独立的测试、加 lint 规则、跑 CI。线上事故少多了。
**但是：** Agent 写了一个大功能，CI 最后才跑，一次报了十几个问题。让 Agent 改，改了这个坏了那个，来回折腾比重写还慢。问题不是验证本身，而是验证来得太晚了。Agent 已经基于一个错误的假设写了五百行代码，这时候再告诉它假设错了，它只能在错误的地基上打补丁。
**我想知道：** 能不能让反馈在它写的过程中就发生，而不是等做完再秋后算账？

---

## 卷一演进：spec 和验证过时了，我自己发现、自己改

> ⚠️ 待采访充实。采访时注意区分 spec 腐烂（文档跟不上代码）和验证腐烂（测试跟不上需求变化），两者解法不同。

**已经：** spec 体系和验证闭环都建好了，跑了一两个月，产出一直很稳。产品也上线了，开始加新 feature，代码不断在变。
**但是：** Agent 的产出质量又开始下降了。一开始以为是 Agent 的问题，查了半天才发现——我上个月写的 spec 里还在引用一个已经废弃的接口，测试里断言的也是老的返回格式。我试过每次全部重写 spec——太贵了。又试过来一个需求就在老 spec 上打补丁——补丁打多了 spec 自相矛盾。代码一直在变，但我建的这套体系是静态的。
**我想知道：** 怎么让 spec 和验证跟着代码一起活着，而不是每隔一两个月就烂掉？

> **这次演进的性质：** 小张还在 human-in-the-loop 阶段，他自己能发现 spec 过时了，自己去改。本质是文档维护问题——怎么低成本保持 spec 和代码同步。

# 卷二：从 5-10x 到几十倍 —— 个人进阶

> **已经：** 单次交互很高效了，spec 写得清楚，验证也闭环了，演进节奏也在摸索。
> **但是：** 每个任务都需要我实时在场，我成了瓶颈。我一天能盯三四个任务就到极限了，Agent 明明可以做更多。
> **我想读完这一卷后能：** 让 Agent 自主跑、多个 Agent 并行，自己从操作者变成分配者，个人产出突破 10x。

---

## 阶段三：放手让它自己跑

> 小张在这个阶段的状态：单次交互已经很高效了，验证闭环也建了，甚至演进的节奏也开始摸索出来了。**但他发现自己成了瓶颈。** 每个任务都需要他实时在场——给输入、盯过程、收结果、触发下一步。Agent 的速度已经不是问题了，他自己的带宽才是。他一天能盯三四个任务就到极限了，但 Agent 明明可以做更多。他想让 Agent 自己跑一段时间，自己去干别的事，回来收结果。

### 3.1 做着做着就乱了

**已经：** 单次交互效率很高了，迭代流程也跑通了。现在我想给 Agent 一个稍大的任务，让它自己跑，我去干别的。
**但是：** 做到一半它开始忘之前的决策、重复已经做过的事、甚至覆盖自己前面写的代码。任务越长越严重。一个需要改四五个文件的功能，整个丢给它会乱；我试过把所有相关文档都塞给它帮它理解，结果更乱——它开始在不相关的地方做修改，把文档里的示例代码当成了真实代码。
**我想知道：** 一个 session 里它能记住多少东西？任务该怎么拆、上下文该怎么给，才能让它在一次执行中不乱？

### 3.2 下次又是新实习生

**已经：** 掌握了任务拆分和上下文管理，Agent 在单个 session 内表现很好了。
**但是：** 上一轮对话里它学到的东西——项目的架构偏好、踩过的坑、上次做到哪了——全部归零。新 session 开始又是一个什么都不知道的新实习生。有时候 session 中途断了，之前的进度也全丢，得从头来。我每次都要花十几分钟重新把背景喂给它。
**我想知道：** 有没有什么机制让 Agent 的知识跨 session 保留？或者至少让下一个 session 能从上一个的进度接着干？

---

## 阶段四：一个人指挥一个舰队

> 小张在这个阶段的状态：已经能让单个 Agent 在合理的任务粒度内自主跑了。他开始想：能不能同时跑好几个？前端一个、后端一个、测试一个。

### 4.1 互相覆盖

**已经：** 单个 Agent 自主执行很顺利了。我开始同时开两三个 Agent，分别做前端和后端。
**但是：** 两个 Agent 改了同一个文件，后提交的把先提交的覆盖了。或者一个 Agent 改了数据库 schema，另一个还在用老的字段名。
**我想知道：** 多个 Agent 并行的时候，怎么防止它们互相踩脚？

### 4.2 各自对但合起来炸

**已经：** 给每个 Agent 划了清晰的文件/模块边界，不再互相覆盖了。
**但是：** 每个 Agent 的产出单独跑测试都没问题，合在一起就炸了。前端期望的 API 返回格式跟后端实际返回的不一样，两边各自的假设是矛盾的。而且管 3 个 Agent 还能靠自己协调，一多就完全分不清谁在干什么、问题是哪个 Agent 引入的。
**我想知道：** 怎么在它们各自开发的时候就保证最后合得上？怎么在出问题时快速定位是谁的锅？

---

## 卷二演进：我不在场了，spec 烂了没人发现

> ⚠️ 待采访充实。

**已经：** 多 Agent 并行跑起来了，我不再实时盯每个 Agent 了。
**但是：** 以前 spec 过时了我自己能发现——因为我在场。现在我不在场了，spec 烂了没人发现，Agent 就一直按过时的指令跑，产出一堆看似正常但其实基于错误假设的代码。等我回来收结果的时候才发现全偏了。
**我想知道：** 我不在场的时候，怎么让系统自动检测 spec 和代码之间的漂移？有没有什么机制能在 spec 过时的时候自动报警，而不是等我人肉发现？

> **这次演进的性质：** 小张已经不在 loop 里了，human-in-the-loop 的维护方式失效。本质是监控和自愈问题——需要自动化机制检测体系与代码之间的漂移。

# 卷三：重塑团队 —— 组织变革

> **已经：** 我个人效率上来了，团队里也有几个人开始用 Agent 了。
> **但是：** 团队的分工、流程、协调机制都是为人的速度设计的，成了新的瓶颈。
> **我想读完这一卷后能：** 知道怎么调团队结构、改流程、定义新角色，让整个团队适配 Agent 时代。

> **视角切换：这一卷的主角是老王**（Tech Lead）。

---

## 5.0 以前靠人协调的事，现在没人管了

> 这不是一条用户故事，而是老王进入这一卷时需要建立的一个核心认知。

以前团队的协作很大程度上靠**人和人之间的非正式沟通**：站会上提一嘴"我在改支付模块"，走廊里碰到聊两句"那个接口我下午要改，你先别动"，code review 时顺便传递架构意图。这些非正式协调几乎零成本，而且人类自带"听到之后记住、下次注意"的能力。

Agent 没有这些。张三的 Agent 不会去"听"李四的 Agent 在干什么。它们各自忠实地执行各自的指令，完全不知道彼此的存在。以前靠人际网络自然解决的协调问题，现在变成了需要**显式设计**的工程问题。这是卷二（个人多 Agent）问题在团队层面的放大，但性质变了——个人的多个 Agent 你还能自己协调，不同人的 Agent 之间连协调的人都没有。

---

## 阶段五：重塑团队

### 5.1 团队结构对不上了

**已经：** 团队里有两三个人已经能熟练使用 Agent，个人产出明显提升。
**但是：** 团队还是按老的方式分工——前端 2 人、后端 3 人、QA 1 人。会用 Agent 的人产出是其他人的好几倍，但分工和绩效还是按人头算。团队内部开始出现不平衡，不会用的人觉得焦虑，会用的人觉得自己吃亏。
**我想知道：** Agent 时代的团队应该怎么分工？人数配比、角色定义要怎么调？

### 5.2 流程跟不上 Agent 的速度

**已经：** 开始在团队层面推广 Agent 使用，更多人在用了。结构问题还没完全理清，但产出已经在加速。
**但是：** Agent 一天 20 个 PR，但我们的 review 流程是为"人一天写 1-2 个 PR"设计的。部署窗口一天两次。PR 堆积、review 排队、合并冲突频发。代码能秒出，但组织流程成了瓶颈。
**我想知道：** 哪些流程需要改？改成什么样？

### 5.3 不同人的 Agent 之间没有协调

**已经：** 调整了一些流程，团队 Agent 使用率上来了。
**但是：** 张三的 Agent 重构了一个接口，李四的 Agent 还在用老接口。以前张三会在站会上说一句"我今天要改支付接口"，李四就知道先别动相关代码。现在 Agent 和 Agent 之间没有这种协调——它们甚至不知道彼此的存在。这不是技术问题，是以前靠人际沟通自然解决的事，现在需要用机制替代。
**我想知道：** 怎么在团队层面建立 Agent 之间的协调机制？以前靠人自然做到的事，现在用什么替代？

### 5.4 我们的角色和绩效怎么定义

**已经：** 团队结构和流程开始适配 Agent 了。
**但是：** 团队里的高手现在几乎不"写"代码了——他们设计 spec、搭建验证体系、管理 Agent 舰队。但他们的 title 还是 Senior Engineer，JD 写的是"设计和实现软件系统"，绩效考核还在数代码产出。
**我想知道：** 这些人的角色应该叫什么？绩效怎么衡量才合理？

---

## 卷三演进：多人各自建的体系互不兼容

**已经：** 团队已经基本完成了 Agent-native 的转型，多个团队成员各自建了自己的 spec 体系、验证流程、Agent 工作流。
**但是：** 这些体系之间开始不兼容。张三的 spec 格式跟李四的完全不同，两个人的 Agent 对同一个模块的假设互相矛盾。以前一个人的体系烂了只影响自己，现在影响整个团队。而且如果代码本身变得容易生成，我们花了几个月建的这些体系——是只对我们有价值的内部资产，还是别人也能很快复制？
**我想知道：** 怎么在组织层面统一和治理这些体系？在 Agent 时代，团队应该积累什么样的资产才是真正的护城河？

> **这次演进的性质：** 不再是一个人的体系过时了，而是多个人各自建的体系之间不兼容。本质是组织治理问题——需要在团队层面定义标准、统一规范、管理体系间的一致性。

```
  卷二演进：我不在场了，spec 烂了没人发现
    → 自动化监控和漂移检测问题 ⚠️ 待采访

━━━ 卷三：组织变革 ━━━

  5.0 以前靠人协调的事，现在没人管了 → 认知铺垫
        ↓
  阶段五：重塑团队
    5.1 团队结构对不上了 → 重新分工
    5.2 流程跟不上速度 → 流程再造
    5.3 不同人的 Agent 之间没有协调 → 用机制替代人际沟通
    5.4 角色和绩效怎么定义 → 新的角色体系
        ↓
  卷三演进：多人各自建的体系互不兼容
    → 组织治理和护城河问题
```

## 演进主题说明

演进是贯穿全书的递归主题，作为每卷的独立子级出现。三次演进性质完全不同：

| 出现位置 | 性质 | 核心问题 | 解法方向 |
|----------|------|----------|----------|
| 卷一末 | Human-in-the-loop | spec 烂了我发现、我去改 | 文档维护：低成本保持 spec 和代码同步 |
| 卷二末 | 自动化 | 我不在场了，spec 烂了没人发现 | 监控与自愈：自动检测 spec 与代码的漂移 |
| 卷三末 | 组织治理 | 多人各自建的体系互不兼容 | 标准化与治理：组织层面的体系统一 |

> ⚠️ 第一次和第二次演进目前为占位故事，待采访充实。采访时注意区分 spec 腐烂（文档跟不上代码）和验证腐烂（测试跟不上需求变化），以及 human-in-the-loop 维护和自动化维护的实践差异。

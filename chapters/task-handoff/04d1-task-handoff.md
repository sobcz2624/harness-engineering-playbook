# 任务没跑完，进度去哪了：任务级交接文档

## 记忆的三个维度

跨会话的记忆不是一个单一的问题，而是三个不同维度的问题。

**角色维度：我是谁。** Skill 卡片回答这个问题。Agent 的职责边界、行为策略、升级规则，这些信息跨项目持久存在。一个 Coder 角色的 Skill 卡片不会因为从支付项目切换到账单项目而改变。

**任务维度：这个任务做到哪了。** 一个具体任务的执行进度、中途做的设计决策、发现的阻塞。这些信息只在任务生命周期内有意义，任务完成即归档。

**项目维度：这个项目里学到了什么。** 架构决策的历史原因、模块的已知脆弱点、踩过的坑。这些知识跨任务持久存在，是整个项目的知识资产。

三个维度相互独立。Skill 卡片不记录任务进度，交接文档不定义角色边界，项目知识库不追踪某个具体task 完成了没有。混淆维度会导致信息错位：把任务进度写进 Skill 卡片，下一个任务启动时就会读到上一个任务的残留状态；把角色定义写进交接文档，任务完成归档后角色信息也跟着消失。

Skill 卡片已经在上一节讨论过。项目级知识的持久化将在后文讨论。本节聚焦任务维度：当一个任务跨越多个会话时，如何让执行进度在会话边界存活。

## 为什么任务级容错不可省略

第 04b 节讨论了任务分解：把大任务切成在上下文崩塌之前能完成的块。理想情况下，每个块都能在单次会话内完成。但这个理想的失败概率远比直觉预期的高，原因来自三个独立的方向。

**上下文消耗不可预测。** 一个自包含的任务规约定义了 Agent 需要读取的信息范围。但执行过程中，Agent 可能读取规约未明确要求的外部文档——为了理解一个接口的行为去查看它的实现，为了确认一个数据格式去翻阅上游模块的代码。这些额外的文件读取是合理的工程行为，但每一次读取都在消耗上下文窗口。当外部文档的实际长度超出预期，或 Agent 的探索路径比预期更长时，一个本应在窗口容量 60% 处完成的任务可能在 90% 时仍未结束。

**模型输出天然不确定。** 大语言模型基于概率生成。同一个任务、同一份输入，十次执行中可能有九次顺利完成，一次 Agent 走了弯路——尝试了不必要的方案、重复解决已解决的问题、或在某个分支上过度探索。这种不确定性是模型的本质属性，不是 bug。你无法通过更好的规约消除它，只能通过容错机制承接它。

**供给侧不稳定。** API 提供商的服务质量不是常量。模型可能在高峰期被静默降级，实际可用的上下文窗口可能小于文档标称的长度，速率限制可能在长任务的中途触发。这些因素在你设计任务分解时完全不可控，但它们直接影响每个任务块是否能在单次会话内完成。

三个因素相互独立，意味着单次任务执行未能在一个会话内完成，不是小概率的极端情况，而是需要被工程化应对的常态风险。

更关键的是，一旦上下文接近饱和，大部分框架和工具会触发**自动压缩**——对历史上下文进行摘要或截断以腾出空间。压缩是一个不可逆的信息损失过程。04a 描述的上下文之墙是产出质量的缓慢下降，而压缩是一次性的断崖：早期的设计决策、接口约定、已完成工作的细节被压缩算法不可预测地丢弃。压缩之后，Agent 的行为确定性无法保证。此时继续执行，不如保存进度、开始新会话。

最后一个考量是任务在整体流程中的位置。在更大的粒度上——比如一个开发阶段完成后——人类可以介入做高质量的闸门检查、交叉验证、手动修正。阶段之间有足够的缓冲吸收偏差。但在任务粒度上，单个任务的执行时间短、任务之间的依赖链紧密。一个任务卡壳或产出偏差，后续依赖它的任务会立刻在错误的基础上继续推进。等到阶段结束时人类介入检查，偏差已经经过多个任务的放大，修复成本远高于在任务边界就截住。任务级的容错不是奢侈品，而是防止小偏差演变为系统性偏离的最后一道内环防线。

任务分解处理的是常态——大多数时候你能切对。交接文档处理的是意外——当你切错时，已经完成的工作不应该白费。两者是互补关系：分解是预防，交接是容错。这和第三章的逻辑一致：测试先行是预防（03a），持续反馈是纠偏（03d），两者共同构成闭环。

## 交接文档：结构与设计原则

一份交接文档需要让一个**完全没有前序上下文的 Agent** 能够接续任务。但"接续"不等于"照搬上一个 session 的思路继续"。新 session 的价值恰恰在于它是一个独立的判断者——它应该基于事实重新规划，而非继承前任的路径。

这引出了交接文档的核心设计原则：**记录事实，不记录推断。**

交接文档中的信息分为两类。事实类信息描述已经发生的事：哪些代码已经写了，做了什么选择，发现了什么阻塞。这些信息可以对照实际代码和测试进行验证。推断类信息是上一个 session 对未来的判断：下一步应该做什么，建议用什么方案。这些信息无法验证，只能被信任或质疑。

如果上一个 session 的执行路径本身就有偏差——比如它选择了一个低效的实现方式，或者对某个模块的理解有误——推断类信息会把这个偏差编码进交接文档。新 session 加载交接文档后，看到了一份看起来合理的"下一步计划"，很可能直接沿着这条路径继续，丧失了独立纠偏的机会。这是第三章讨论的共谋问题（03b）在会话间的变体：上一个 session 执行任务又规划后续，自己给自己定方向，错误得以跨会话传播。推断类信息也是上下文污染的一个来源（04c 推理污染）：前一个 session 的推理产物进入新 session 的上下文，干扰独立判断。

因此，交接文档应该只包含事实类字段。新 session 拿到事实后，结合原始规约（Ring 0 信任源，02e），自行推导下一步行动。

一份最小的交接文档包含以下结构：

```
spec_ref: "/specs/xxx.md line 42-56"

status:
  completed:
    - "JWT 签发逻辑（RS256）— 已通过单元测试"
    - "登录端点 /api/login — 已通过集成测试"
  in_progress:
    - "Token 刷新端点 /api/refresh — 签发完成，吊销逻辑未开始"
  pending:
    - "登出端点 /api/logout"
    - "Token 黑名单机制"

decisions:
  - what: "选择 RS256 而非 HS256"
    why: "需要跨服务验证，非对称签名允许服务只持有公钥"
  - what: "refresh token 存数据库而非 Redis"
    why: "当前阶段用户量不需要 Redis，降低基础设施复杂度"

blockers:
  - "user 表缺少 refresh_token_hash 字段，需要先跑 migration"
  - "发现现有 session 中间件和 JWT 逻辑有冲突，需确认是否替换"

files_changed:
  - "src/auth/jwt.py — 新建"
  - "src/auth/login.py — 新建"
  - "tests/auth/test_jwt.py — 新建"
  - "tests/auth/test_login.py — 新建"
```

注意这份文档中没有 `next_steps` 字段。这是刻意的设计选择。

每个字段的设计意图：

**spec_ref** 指向原始规约。交接文档只记录执行状态，不重复规约内容，避免两份文档之间出现不一致（02f 规约腐烂的教训同样适用于交接文档）。新 session 应始终以原始规约为权威信息源，交接文档为辅助信息源。当两者冲突时，以规约为准。

**status** 用三态（completed / in_progress / pending）而非简单的 checklist。关键区别在于 in_progress：它标记的是"做了一半"的工作，这正是会话中断最容易丢失的信息。一个 subtask 要么完全没做（pending），要么完全做完了（completed），都好处理。做了一半才是最危险的——新会话如果不知道哪些部分已经实现、哪些还没有，可能重复实现已有逻辑，或者在半成品上继续构建导致不一致。

**decisions** 记录的是"做了什么选择"和"为什么做这个选择"。这里的 why 看似也是推理产物，但它与 next_steps 有本质区别：decisions 描述的是**已经发生的选择**，新 session 面对的是一个已存在的代码事实（代码已经用了 RS256）。why 帮助新 session 判断"这个选择在当前条件下是否仍然合理，是否需要推翻"，而非指示"下一步该往哪走"。它是回顾性的解释，不是前瞻性的指令。这和隐性知识外化（02d）是同一个原则：未被记录的决策理由等于不存在。

**blockers** 是显式的阻塞信号。人类程序员会自然记住"昨天卡在那个 migration 上了"，Agent 不会。如果 blocker 没有被写下来，新会话可能直接跳过这个阻塞继续实现，产出的代码在集成时必然失败。

**files_changed** 提供变更的事实清单。新 session 可以据此检查实际代码状态，验证 status 中的声明是否属实，而非盲目信任交接文档的描述。

## 交接文档的触发时机

需要先明确一个前提：交接文档不是常规流程的一部分。如果一个任务在单次会话内顺利完成，不需要交接文档——任务产出直接进入后续的验证和审查流程。交接文档只在任务**未能在单次会话内完成**时产生。它是一个中间状态的快照，是任务分解容错机制的产物。

既然交接文档只在 session 被迫中断时产生，问题就不是"什么时候写"，而是**如何触发写入**。核心挑战在于：Agent 目前普遍缺乏对自身上下文消耗的感知能力。它不知道自己的窗口还剩多少空间，也不会主动判断"我该保存进度了"。触发必须来自外部。

**Hook 触发。** 部分工具框架支持在特定事件（如 token 用量超过阈值）时执行预定义的操作。通过配置 hook，可以在上下文消耗达到预设比例时，向 Agent 注入一条指令：停止当前工作，按交接文档的结构保存进度。这种方式与工具框架耦合，但执行确定性高。

**外部监测触发。** 一个独立的监测进程持续跟踪 Agent 的 token 使用量。当使用量接近上限时，监测进程向 Agent 注入 prompt，告知它需要开始写交接文档。这种方式不依赖特定工具框架的 hook 机制，适用范围更广，但需要额外的基础设施。

**原生感知（未来）。** 如果 Agent 本身具备上下文消耗的感知能力，它可以在执行过程中自主判断何时保存进度，无需外部触发。这是最自然的方式，但依赖模型和工具框架的能力演进。

无论哪种触发方式，关键约束是：**触发必须发生在自动压缩之前。** 一旦进入自动压缩，历史上下文被不可逆地截断或摘要，Agent 对已完成工作的记忆已经不完整。此时写出的交接文档质量无法保证。触发阈值应留出足够的窗口空间，让 Agent 在上下文仍然完整的状态下完成交接文档的写入。

交接文档写入后，当前 session 终止。新 session 启动时加载原始规约和交接文档，基于事实信息独立判断下一步行动，继续执行未完成的部分。任务最终完成后，交接文档作为过程归档保留，供后续复盘参考。

## 交接文档的固有风险

交接文档的写入场景天然带有风险：它只在上下文接近饱和时产生，而这恰恰是 Agent 状态最不理想的时刻。

**不准确。** Agent 在上下文拥挤时写的交接文档，可能遗漏关键信息或错误总结已完成的工作。一个被标记为 completed 的 subtask 实际上只完成了 80%，新 session 跳过它继续推进，最终集成时才发现缺口。缓解方式：交接文档中 completed 的条目应有可验证的证据（如"已通过单元测试"），而非仅靠 Agent 的自我报告。新 session 启动时也应将交接文档中的状态声明与实际代码和测试结果做比对，而非盲目信任。

**过时。** 交接文档反映的是写入时刻的状态。如果人类在两个 session 之间手动修改了代码或调整了规约，交接文档就和实际状态脱节了。缓解方式与上条相同：新 session 应以实际代码状态为准，交接文档为参考。

**膨胀。** 如果同一个任务经历了多次 session 中断，交接文档可能累积大量已完成的历史条目。这些历史信息占用上下文窗口，但对当前执行的指导价值递减。缓解方式：每次更新时精简已完成的条目，只保留对当前执行有直接影响的信息。

# 围绕治理而非执行重新设计角色

传统团队的角色围绕执行层设计：谁写前端代码、谁写后端代码、谁写测试、谁做运维。当 Agent 接管了执行层，这些角色的定义需要从执行层上移到治理层：谁定义做什么、谁编排怎么做、谁验证做得对。

目前行业里有两种主要的探索方向。

**头狼模型。** 黄东旭在 PingCAP 的实践代表了这个方向：一个顶级工程师端到端负责一个产品或模块。他同时扮演产品经理、架构师、主程序员和测试负责人的角色，带领一群 Agent 在自己的领地里工作。他 90% 的时间花在评估 AI 的工作成果上，而非写代码。

头狼模型的优势在于决策链极短。一个人拥有完整的上下文，不需要跨角色沟通，不需要等待审批，可以以极高的速度迭代。黄东旭用这个模式将 TiDB 的 PostgreSQL 兼容层重写到了接近生产水平。

头狼模型的瓶颈在于人才。这个角色需要产品感、架构能力、编码经验和质量意识的综合体。符合条件的人凤毛麟角，而且这样的人往往更倾向于自己创业而非为别人工作。大多数公司的大多数团队，找不到也留不住这样的人。

**Product Tri-Ownership 模型。** 社区提出的另一个方向是将超级个体的能力拆分为三个可培养的专业角色。这个思路借鉴了建筑行业：建筑师定义空间，土木工程师设计结构，监理独立验收，施工方负责执行。在 Agent 时代，施工方就是 Agent。

Product Owner 负责做什么：定义产品愿景、管理用户故事和验收标准、决定优先级。对产出的价值负责，而非对产出本身负责。

Tech Owner 负责怎么做：详细设计、Agent 工作流编排、工具选型。核心职责是设计和优化多个 Agent 的协作方式。不同的任务类型需要不同的工作流，bug fix 流程不同于新功能流程，Tech Owner 负责为不同任务设计合适的工作流。

Quality Owner 负责做得对不对：设计质量流程、全生命周期的质量管控、独立验证。Quality Owner 的存在直接解决了共谋问题：产出代码的 Agent 和验证代码的 Agent 由不同的人管理，利益不对齐，形成天然的对抗。

PTO 模型的优势在于可复制。三个角色中的每一个都比超级个体容易培养。劣势在于需要组织变革，需要三个人紧密协作的默契，以及角色之间的边界需要磨合。

两个模型之外，还有更温和的培训模式：不动组织结构，由一个有经验的人定制 skills、agents 和 workflows，让团队成员按流程执行。门槛最低，但价值天花板也最低。

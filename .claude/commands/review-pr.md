审阅 PR 的写作质量。

第一步：使用 `gh pr list` 列出当前所有 open 的 PR，展示给用户，询问用户要审阅哪一条。

第二步：用户选择后，使用 `gh pr diff <编号>` 获取 PR 内容，按以下六条原则逐条审阅。

第三步：审阅结果先展示给用户确认。绝对不要未经用户确认就发送评论。用户确认后再使用 `gh pr comment` 发到 PR comment。

---

## 审阅原则

### 0. Takeaway 是否明确

读者读完能否用一句话说出这一节的核心信息？如果 takeaway 不明确或者同时有多个互相竞争的 takeaway，指出具体是哪些内容在互相竞争，建议作者先明确一个核心 takeaway。

### 1. 整体 Flow：问题驱动

列出文章的实际 outline（每一节在做什么），从读者视角检查 flow 是否顺畅。核心标准：每一节是否从读者能共鸣的问题开始，解法是否跟在问题后面。如果文章上来就给方案、给分类、给原则，而读者还不知道这在解决什么问题，指出具体的断裂点。

### 2. Why 驱动下降

从抽象到具体的每一步是否被 motivate。作者不能 assert 一个结构、模板或原则然后期待读者接受。每一个具体实践的引入，都要先让读者感受到"如果没有这个会怎样"。如果文章中出现了"从天而降"的方案（读者会问"为什么是这个？凭什么？"），指出具体位置。

### 3. 节间因果递进

上一节的解法是否自然引出下一节的问题。如果多个小节之间是并列堆砌（A、B、C 三个独立话题），而非因果递进（A 解决了问题 X，但引出了问题 Y，Y 由 B 解决），指出具体哪些节之间缺乏因果关系。

### 4. Abstraction Level 一致性

同一层级的节是否保持相近的抽象程度。整体是否从高到低递进。如果出现了层级跳跃（从原则直接跳到 hook 配置、从设计模式跳到具体 API 调用），指出具体位置并建议精简或降级为附注。

### 5. 可读性

是否使用了大量 bullet point 或加粗标题做 mini-section。是否符合项目的语言风格（务实、理性、克制，用自然语言段落表达）。可读性问题指出即可，注明这些由编者统一修改，不需要贡献者在这个阶段花时间打磨。

## 输出格式

审阅结果按以下结构组织：

1. 先肯定贡献的价值和方向（一两句话）
2. 按原则 0-5 逐条给出具体反馈，引用文章中的原文说明问题
3. 结构性问题（0-4）建议在合并前讨论对齐
4. 可读性问题（5）注明由编者后续统一处理
5. 指向项目的写作方法论（CLAUDE.md 中的写作流程）作为参考

语气要求：肯定贡献者的投入和想法，指出的是结构性问题而非否定内容本身。社区共建项目，贡献者是义务劳动，尊重他们的时间和热情。

## Review 自身的语言要求

审阅评论本身的写作必须遵循以下风格：

务实、理性、克制。用思考深度体现专业，不堆砌宏大词藻，不用文学性比喻。

- 不用华丽辞藻，不用"惊喜"这类营销词汇
- 不滥用 bullet points，尽量用自然语言段落表达
- 不说废话，不说客套话，直奔主题
- 用数据和逻辑说话，不靠形容词
- 不要用破折号（——/—/--）。能拆成两句的，拆开写；能用冒号或分句表达的，用冒号或分句
- 避免否定句式，改用正向陈述。与其说 X 不是 Y，不如直接说 X 是什么

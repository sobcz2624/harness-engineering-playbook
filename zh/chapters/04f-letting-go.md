# 从陪聊到放手：执行模式的转变

把本章的内容组合起来，你的工作方式发生了质的变化。

之前，你是 Agent 的实时对话伙伴。你输入一段需求，Agent 产出一段代码，你检查，给反馈，Agent 修改，你再检查。你的时间和 Agent 的时间一比一绑定。Agent 的生产力上限就是你的注意力上限。

现在，你是任务的设计者和验收者。你的新工作流是：将项目拆成模块，为每个模块写 spec 和验收标准，设定阶段闸门，然后放手。Agent 自主执行中间过程。它在围栏内工作，按小步推进，每步通过测试验证。会话结束时知识被持久化，下次会话从上次的进度继续。你在关键节点介入验收，其余时间可以做其他事。

这个转变的核心在于：你的时间不再和 Agent 的执行时间绑定。Agent 可以在你睡觉的时候继续推进项目。你早上起来看到的是一组已经通过测试的代码变更，而非一个等待你输入的对话窗口。

这个模式下，你的产出可以提升一个数量级。之前你一天可能完成一两个功能模块。现在你一天可以设计和分解十个任务块，Agent 在接下来的时间里逐个执行。你的瓶颈从执行速度变成了任务设计的速度。

但这个模式也有前提条件。规约必须足够清晰（第二章）。验证体系必须足够可靠（第三章）。任务分解的粒度必须合适，上下文工程必须到位，知识持久化机制必须运转（本章）。缺少任何一环，放手就意味着失控。这就是为什么本书按这个顺序展开：每一章都是下一章的前提。

# 瓶颈转移：从代码到组织

传统软件团队的瓶颈在执行层：代码写得不够快，功能交付不够多。解法是加人。每个新工程师带来增量的编码能力，团队产出随人数增长。

Agent 让执行变得廉价。一个工程师加上 Agent，编码产出可以提升一个数量级。但指挥和验证的成本没有同步下降。定义需求仍然需要人类理解业务。设计架构仍然需要人类的工程判断。验证产出是否符合意图仍然需要人类的审查。这些工作无法简单地交给 Agent。

瓶颈从"写代码"转移到了"组织人类"。团队的限制因素不再是有多少人在写代码，而是人类之间怎么组织自己来指挥和验证 Agent 的产出。

PingCAP CTO 黄东旭在实践中观察到一个更深层的现象。顶级 vibe coder 各自的工作流和最佳实践并不相同。每个人都发展出了自己的一套指挥 Agent 的方式：不同的 prompt 策略、不同的验证流程、不同的工具组合。两个顶级 vibe coder 很难在同一个模块中协作，"同一片领地里很难容纳两匹头狼，会造成 1+1 < 2"。

这意味着传统的"人多等于产出多"的线性扩展模型彻底失效了。不仅仅是因为 Agent 改变了产出速度，更因为人类指挥 Agent 的方式高度个性化，难以标准化和组合。

组织设计的挑战因此变成了：如何让多个人类各自高效地指挥自己的 Agent 军团，同时确保他们的产出能集成为一个一致的系统？

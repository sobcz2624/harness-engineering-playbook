# 搭跑道再放飞机：平台工程先行

你开始搭建隔离机制、编写 API Contract、配置集成测试。你发现：在多个 Agent 开始写业务代码之前，有大量基础设施需要先就位。

CI/CD 流水线必须能处理多分支并行构建和频繁的集成测试。测试框架必须支持模块级测试和跨模块的集成测试。代码仓库的分支策略需要适配多 Agent 并行的工作模式。监控和告警需要能区分不同 Agent 的产出并定位问题来源。

这些基础设施的搭建本身就是一个项目。如果在 Agent 已经开始并行工作之后再补建，你会陷入一边灭火一边铺路的困境。平台工程必须先于业务代码存在。在放出第一个并行 Agent 之前，跑道必须铺好。

平台的稳定性直接决定了多 Agent 协作的可靠性。如果 CI 经常假阳性（测试随机失败），你无法区分是 Agent 的代码有问题还是 CI 本身不稳定。如果集成测试执行时间太长，反馈循环变慢，问题在被发现之前已经扩散。如果监控缺乏粒度，你无法追溯某个问题是哪个 Agent 在哪个任务中引入的。

社区的实践经验是：先投资平台工程，再扩大 Agent 规模。急于增加并行 Agent 数量而基础设施跟不上，结果是产出速度提上去了但质量下来了，总体效率可能还不如少几个 Agent 但基础设施稳定的状态。
